### 后端分模块说明

`VibeCampus-Backend` 为 **Maven 聚合工程**（根目录 `pom.xml` 的 `packaging` 为 `pom`），子模块统一前缀 `vibecampus-*`，业务代码根包名为 **`cn.ayeez.vibecampus`**。PC 端（Vue）与小程序（UniApp）共用同一套 **REST**，建议接口路径统一挂在 **`/api/v1`** 下（各模块再细分路径，例如用户相关在 `/api/v1/user`）。

| 模块                       | 职责概要                                                                                                              |
| ------------------------ | ----------------------------------------------------------------------------------------------------------------- |
| **vibecampus-bootstrap** | 应用入口：包含 `VibeCampusApplication`、全局配置（如 `application.yml`）、Spring Security 等横切能力；依赖并组装其余业务模块，是唯一需要执行 `main` 启动的模块。 |
| **vibecampus-common**    | 各业务模块共享的类型与工具（如统一返回体、错误码、通用工具类等），避免在业务模块间复制粘贴。                                                                    |
| **vibecampus-user**      | 用户与账号体系（注册登录、个人资料等）。                                                                                              |
| **vibecampus-post**      | 帖子与内容相关（发帖、分类、点赞等）。                                                                                               |
| **vibecampus-comment**   | 评论与回复。                                                                                                            |
| **vibecampus-admin**     | 管理端接口（与普通用户接口同属一套 REST，通过路径与权限区分，例如 `/api/v1/admin`）。                                                             |

每个业务模块（`user` / `post` / `comment` / `admin`）内部采用 **包分层**，约定为：

- **`...*.controller`**：对外 HTTP 接口，只做参数校验与调用 Service。
- **`...*.service`**（实现类可放在 **`...*.service.impl`**）：业务逻辑。
- **`...*.mapper`**：MyBatis 数据访问接口。

MyBatis 的 `@MapperScan` 集中在启动类上，扫描上述各模块的 `mapper` 包。日常开发时：**在对应业务子模块中写代码**，需要改端口、数据源等环境配置时，按 **Spring Profile** 使用 `vibecampus-bootstrap/src/main/resources/` 下对应文件（见下文「后端运行环境与配置文件」）。

在聚合工程根目录 `VibeCampus-Backend` 下也可用命令行启动：`mvn -pl vibecampus-bootstrap spring-boot:run`（需已安装 Maven 并配置好 `JAVA_HOME`；默认 `local` Profile，须已有 `application-local.yml`）。若暂未创建本地文件、只想快速起服务，可临时指定：`mvn -pl vibecampus-bootstrap spring-boot:run -Dspring-boot.run.arguments=--spring.profiles.active=test`。

4. **配置文件（三环境）**：见下 **「后端运行环境与配置文件」**。本地开发须复制示例生成 `application-local.yml`（该文件已被 `.gitignore` 忽略，勿提交）。

#### 后端运行环境与配置文件

配置文件位于 `vibecampus-bootstrap/src/main/resources/`，按 **Spring Profile** 分为三套（公共项在 `application.yml`，各环境在对应 `application-{profile}.yml`）：

| Profile | 文件 | 用途与敏感信息 |
| ------- | ---- | -------------- |
| **local** | `application-local.yml`（仅本机，**不入库**） | 开发人员本机数据库密码、本机 Redis 等。首次请复制同目录下的 **`application-local.example.yml`** 为 `application-local.yml` 后按需修改。 |
| **test** | `application-test.yml`（**提交到 Git**） | 测试环境：库连接等**不写死密钥**，通过**环境变量**注入；仓库内可保留非敏感默认值。CI（如 GitHub Actions）在未注入数据库变量时，可使用文件内自带的内存 **H2** 默认项跑通 `mvn test`。 |
| **prod** | `application-prod.yml`（**提交到 Git**） | 生产环境：同样只写变量占位，**必须**由部署平台在运行时注入；未设置 `SPRING_DATASOURCE_URL` 等变量时不应启动成功，避免误连错误库。 |

激活方式：设置环境变量 **`SPRING_PROFILES_ACTIVE`** 为 `local`、`test` 或 `prod`。本地未设置时，工程默认使用 **`local`**（因此必须先准备好 `application-local.yml`）。

**测试与生产**共用同一套变量名（例如 Spring Boot 标准项 `SPRING_DATASOURCE_URL`、`SPRING_DATASOURCE_USERNAME`、`SPRING_DATASOURCE_PASSWORD`），通过**不同部署环境**注入不同取值。建议在 GitHub 上创建两个 **Environments**（例如名称 **`test`** 与 **`production`**），分别在各自环境下配置 **Secrets**（密钥仅保存在 GitHub，不会出现在仓库文件中）；在对应 Workflow 或部署脚本里为 Job 指定 `environment: test` 或 `environment: production`，并把 Secrets 映射为上述环境变量后启动应用（`SPRING_PROFILES_ACTIVE=test` 或 `prod`）。仓库中已提供示例 Workflow：`.github/workflows/backend-ci.yml`（CI 使用 `test` Profile，无需数据库 Secrets 即可完成 `mvn verify`）。

5. **启动服务**：
    - 在 IDE 中打开聚合工程 `VibeCampus-Backend`，运行模块 `vibecampus-bootstrap` 下 `cn.ayeez.vibecampus.VibeCampusApplication` 的 `main` 方法。
    - 控制台输出 `Tomcat started on port 8080` 即代表启动成功。

### 4. 团队协作流程 (Git Flow)

为了保证代码质量与版本稳定性，团队需严格遵守以下 Git 工作流：

1. **分支策略**：
    
    - `main`：生产环境分支，保护分支，严禁直接推送。
    - `develop`：开发集成分支，对应当前开发版本的最新代码。
    - `feature/*`：功能开发分支，基于 `develop` 切出。
    - `hotfix/*`：紧急修复分支，基于 `main` 切出。
2. **开发流程**：
    
    - **切分支**：开始新功能前，从 `develop` 拉取新分支，命名规则为 `feature/模块名-日期`（例如 `feature/user-auth-20260412`）。
    - **编码与提交**：
        - 遵循团队代码规范（Checkstyle/ESLint）。
        - Commit Message 格式：`<type>(模块): 描述`（例如 `feat(user): add login interface`）。
    - **同步代码**：每天开始工作前，先 `git pull origin develop` 更新 `develop` 最新代码，解决可能的冲突。
    - **推送代码**：开发完成后，推送到远程仓库。
    - **发起 PR (Pull Request)**：
        - 目标分支选择 `develop`。
        - 在 PR 描述中详细说明修改内容、涉及的问题单号。
        - **必须**至少有一名其他成员 Code Review 通过后，方可合并。
3. **代码审查 (Code Review) 重点**：
    
    - 是否符合《校园bbs开发技术文档》中的架构设计？
    - 是否存在 SQL 注入、XSS 等安全漏洞？
    - 是否添加了对应的单元测试（后端 JUnit）？

## 分支说明

**目的**：确保 `main` 分支始终处于可发布（Stable）状态，隔离开发中的功能（Unstable），防止未经测试的代码直接上线。
#### 1. 核心分支定义

| 分支名称          | 远程路径             | 用途                      | 部署环境 |
| ------------- | ---------------- | ----------------------- | ---- |
| **`main`**    | `origin/main`    | 存放线上运行的稳定代码。            | 生产环境 |
| **`develop`** | `origin/develop` | 集成所有开发完成的新功能，用于日常联调和测试。 | 测试环境 |

#### 2. 日常开发流程


**步骤 1：开工前先“同步” (避免冲突)**  
在你开始写代码前，或者修改文件前，先拉取最新的代码。因为你的队友可能刚提交了代码，如果你直接写，可能会覆盖别人的成果。
```bash
# 确保你在 develop 分支
git checkout develop

# 拉取远程最新的 develop 代码
git pull origin develop
```


**步骤 2：写代码 & 调试**
- 直接在 `develop` 分支上修改文件。
- **建议**：每次只专注于一个小功能点（比如“改一下登录按钮样式”或“修一下发帖接口”），尽快完成，不要让代码在本地停留太久。


**步骤 3：提交前再“同步” (解决冲突)**  
在你点击提交按钮前，**强烈建议**再执行一次 `git pull`。这是防止代码冲突的最有效手段。
```bash
git pull origin develop
```

- 如果提示 `Already up to date`，说明没人动代码，你可以安全提交。
- **如果提示有冲突（Conflict），请在本地解决冲突，然后重新编译运行，确保没问题后再进行下一步！！！！！！！！**

**步骤 4：推送到远程**  
将你的代码推送到 GitHub。
```bash
git add .
git commit -m "描述你改了什么，例如：修复了登录页的样式溢出"
git push origin develop
```

#### 3. 发布上线流程

当我们要准备上线一个新版本（比如修复了一堆 Bug，或者上线了新功能）时，才需要操作 `main`分支。

**由阿叶Ayeez执行以下操作：**

1. **确保 `develop` 是稳定的**：让大家都把代码推上来，并且经过了简单的联调测试。
2. **合并代码**：
```bash
# 切换到本地 main 分支
git checkout main

# 拉取远程最新的 main (以防万一)
git pull origin main

# 从 develop 合并代码进来
git merge develop

# 推送到远程 main
git push origin main
```

3. **打标签 (Tag)**：  
    在 GitHub 的 Release 页面，基于最新的 `main` 提交打一个 Tag（例如 `v1.0.1`），方便以后回滚。

### 规则怪谈（必须遵守！！！）
为了防止 `develop` 分支炸掉，请遵循这几点：
1. **绝不提交敏感信息**：不要把数据库密码、个人配置文件提交到 `develop`。
2. **绝不强制推送 (Force Push)**：**严禁**使用 `git push --force`。如果 `push` 被拒绝，请先 `pull`。
3. **编译通过才能推**：推送到远程前，确保你的代码在本地 `npm run build` 或后端打包是成功的。不要把一堆报错的代码扔给队友。
